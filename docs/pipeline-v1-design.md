# パイプラインV1 設計書(レビュー用)

版: v1.0 draft(2026-07-20)/ 目的: 長尺1本目「米CPIの翌日、日経平均は過去10回どう動いたか」を実データで制作できる最小パイプライン

---

## 1. 設計思想(5原則)

1. **人間は判断だけ、機械は作業だけ**: 人間の工程は①テーマ選定 ②台本検収 ③公開判断の3点のみ。データ取得・集計・作図・音声・レンダリングは全てコード化する
2. **全数値にトレーサビリティ**: 動画に出る数字は必ず「ソースID+取得日時+加工コード」に遡れる形でDBに保存。画面の出典表示・概要欄・GitHub公開はこのメタデータから自動生成(信頼設計の実装)
3. **冪等・再実行安全**: どのステップも再実行しても壊れない(fetchは差分取得、renderは上書き)。失敗したステップだけやり直せるCLI設計
4. **ローカルファースト・低依存**: サーバー不要。SQLite+ファイルシステム+ローカルレンダリング。有料APIは使わない(LLMのみ従量)。月額コスト目標1,000円台
5. **段階的自動化**: V1は「投稿」を自動化しない(YouTube Studioで手動公開)。品質が安定してからV2でAPI投稿・Shorts日次・X連携を足す

## 2. データソース詳細(API有無・認証・制限)

| データ | ソース | API | 認証 | 制限・備考 |
|---|---|---|---|---|
| 米CPI(値・発表日) | **FRED API**(セントルイス連銀) | ◎ 公式REST | **無料APIキー要**(即時発行) | series: CPIAUCSL等。**releases APIで「発表日程」も取れる**(検証の起点日に必須)。レート制限緩い |
| 米金利・雇用・マクロ全般 | FRED API | ◎ | 同上 | DGS10(10年金利)、UNRATE等。V1はCPIのみ、器は共通 |
| 日経平均・TOPIX・S&P500・SOX・ドル円 | **yfinance**(Yahoo Finance非公式) | ○ 非公式 | 不要 | ^N225 等の日足。**非公式のため: 生データの再配布はせず動画内の図表利用に留める/障害時の代替としてStooq(無認証CSV)を実装** |
| 13F(著名投資家保有) | SEC EDGAR API(data.sec.gov) | ◎ 公式 | キー不要(**User-Agent明示必須**) | 10req/s上限。**V1では未使用(V2のバズ企画用)**。器だけ定義 |
| 先物ポジション | CFTC COT(publicreporting.cftc.gov) | ◎ Socrata API | キー不要 | 週次。V1未使用・V2候補 |
| 経済指標カレンダー | FRED releases API | ◎ | FREDキー | 民間カレンダーサイトのスクレイピングは規約リスクがあるため**使わない** |

- **APIキーが必要なのはFREDのみ**(無料・メール登録で即発行)。`.env`で管理しGitにはコミットしない
- LLM(台本ドラフト): Claude API(従量)。1本あたり数円想定

## 3. アーキテクチャ

```
market-video/                     # TypeScript monorepo(Remotionと言語統一)
├ pipeline/
│  ├ fetch/    fred.ts yahoo.ts stooq.ts edgar.ts  # ソース別アダプタ(共通IF)
│  ├ db/       schema.ts (SQLite: series, observations, events, provenance)
│  ├ analyze/  eventStudy.ts   # イベント日×翌営業日騰落の集計(純関数・ユニットテスト対象)
│  ├ script/   draft.ts        # 集計JSON→LLM→台本Markdown(数値はプレースホルダ埋込で捏造防止)
│  └ cli.ts    # `fetch` `analyze` `script` `render` の4コマンド
├ video/       # Remotion: チャートアニメ・字幕・出典表示のテンプレコンポーネント
├ scripts/     # 検収済み台本(Markdown)置き場 = ここが人間のレビューポイント
└ out/         # レンダリング済みmp4・サムネ・X用図解png
```

### ワークフロー(1本の制作)

```
1. cli fetch cpi nikkei        → FRED/yfinanceから取得、SQLiteへ(出典メタ付き)
2. cli analyze cpi-event       → 「CPI発表日の翌営業日の日経騰落」直近N回を集計→JSON
3. cli script                  → LLMが台本Markdownドラフト生成(全数値は集計JSONから機械挿入)
4. [人間] scripts/xxx.md を編集・検収(誤読・言い回し・固有名詞)
5. cli render                  → VOICEVOX音声→Remotion合成→mp4+サムネ+X図解を out/ へ
6. [人間] プレビュー→YouTube Studioで手動アップロード・公開
```

### 捏造防止の要

台本内の数値はLLMに書かせない。**集計JSONの値をテンプレート変数として機械挿入**し、LLMは「つなぎの文章」だけを書く。これで「AIが数字を間違える」事故を構造的に排除する(金融チャンネルの生命線)

## 4. V1スコープ(作る/作らない)

- ✅ 作る: FRED+yfinanceアダプタ、SQLite、CPIイベントスタディ集計(テスト付き)、台本ドラフト生成、VOICEVOX連携、Remotionテンプレ(オープニング/チャート/検証表/出典/エンディング)、サムネ・X図解生成
- ❌ 作らない(V2+): YouTube API投稿(監査申請だけ先に出す)、Shorts日次自動化、EDGAR 13F、COT、X自動投稿、検収用Web UI(V1はMarkdown編集で足りる)

## 5. リスクと対処

| リスク | 対処 |
|---|---|
| yfinance仕様変更・遮断 | Stooqフォールバック実装。取得層はアダプタ分離で差し替え可能 |
| 祝日・休場日のズレ(翌営業日判定) | 日本の営業日カレンダーをコード内で保持しテストで担保 |
| LLMの数値幻覚 | §3の機械挿入方式で構造的に排除 |
| VOICEVOXの読み間違い | 検収時に読み仮名指定(台本Markdownにルビ記法) |

## 6. レビューしてほしい点

1. データソース構成(特にyfinance非公式利用の割り切りとStooqフォールバック)で問題ないか
2. 人間の関与ポイント(台本検収+手動公開)の位置は適切か、もっと自動/手動に寄せたいか
3. V1スコープの線引き(13F・Shortsを後回し)に異論はないか

## 7. レビュー反映(2026-07-20 改訂)

1. **IPブロック対策(取得層再設計)**: Stooqを主系・yfinanceを補完に逆転/SQLite差分取得で「1日1回・数銘柄・未取得日のみ」に最小化/直列実行+2〜5秒ランダム待機/429・999で指数バックオフ→3失敗でソース単位サーキットブレーカー(24h停止・自動フォールバック)。プロキシ回避は不採用
2. **動画化は検収後フルオート**: 台本をapproved/へ移すと VOICEVOX→Remotion(集計JSON駆動チャート)→mp4+サムネまで自動。生成AI映像(Sora型)は事実性の観点で不使用
3. **スコープ順序変更**: V1=Shorts日次テンプレ(毎日投稿でパイプラインを実地テスト)→V1.5=長尺1本目。収益化時間と信頼構築は長尺が担う方針は不変
