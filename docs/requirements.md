# 要件定義書: 当落ノート(仮称)— 推し活の期限とお金を全部引き受けるアプリ

版: v1.0(2026-07-17)
前提資料: [L2深掘りレポート](./oshikatsu-deep-dive.md) / [日常系・若年層調査](./lifestyle-app-research.md)

---

## 1. プロダクト概要

### ビジョン

> 申込を登録した瞬間に、当落発表・入金締切・発券期限・FC更新の通知がぜんぶ自動でセットされる。
> 「入金忘れで当選無効」を、この世からなくす。

### 解決する課題(優先順)

1. **当落ライフサイクルの管理漏れ**: 多段抽選×複数公演×複数名義で申込が数十件に膨らみ、入金忘れ=即キャンセル+FC信用失墜という実害
2. **使いすぎの不安**: 推し活支出の見えない化(月予算・推し別集計・遠征費)
3. **ツール分断**: スプシ+Trello+カレンダー+紙手帳の併用状態を1アプリに集約

### ターゲットとペルソナ

- メイン: 10代後半〜20代のチケット現場系オタク(旧ジャニーズ・K-POP・2.5次元・アニメ/声優ライブ)
- ペルソナ①「多ステ遠征社会人」(25歳・複数名義・年40万円): 当落マトリクス・期限通知・遠征費
- ペルソナ②「身の丈推し活学生」(19歳・月1万円以下): 予算上限・推し貯金・シェアカード
- ペルソナ③「在宅配信課金型」(22歳・VTuber): 課金累積の可視化・FC/メンバーシップ更新管理 ← MVPでは部分対応

### ポジショニング

- 「デコと記念日」のOshibana、「遠征」の&Fanfun、「支出」の推しPayに対し、**「期限とお金の実務」の決定版**を先に取る
- 転売・譲渡マッチング機能は持たない(転売助長と誤認されない設計を最初から明文化)

## 3. 機能要件(MVP)

### F1. 推し管理

- 推しの登録(名前・ジャンル・推し色2色・アイコン絵文字or画像)。無料: 3人まで / Pro: 無制限
- 推しごとにFC/メンバーシップ登録(名称・更新日・年会費)→ 更新通知(F5)
- 推し色はアプリ全体のアクセントカラーに反映(推しごとに画面が染まる)

### 設計判断: 推し情報の「自動取得」について

「メニューから推しを選ぶと公演・スケジュール情報が自動で入ってくる」体験は理想形だが、**公演情報の網羅的な公開APIは存在せず**、実現には①アーティストマスタDBの構築・保守、②公演情報の継続的キュレーション(またはスクレイピング=規約リスク)、③サーバー運用が必要で、個人開発のMVPには過大。そこで段階的に近づける:

| 段階 | 体験 | 実装 |
|---|---|---|
| MVP | 推しを手入力で登録(名前+色)。公演・申込はユーザーが登録し、**通知の自動化**で価値を出す | ローカル完結・運用費0 |
| v1.1 | 推し名のオートコンプリート(主要アーティストの静的リスト内蔵)+公式サイト/X/FCへのリンク集自動設定 | 静的マスタ(アプリ内蔵、権利リスクなし: 名称はファクト、写真は使わない) |
| v1.2 | **配信・リリースは自動取得**: YouTube/Twitchの配信予定、音楽リリース(公式APIが存在する領域だけ自動化) | 公開API連携 |
| v2 | 公演情報DB: 運営キュレーション+ユーザー投稿の共有(「同じ推しの人が登録した公演を取り込む」) | サーバー+モデレーション。ネットワーク効果でロックイン強化 |

方針: **「情報を運ぶ」のは段階的に、「期限を守らせる」のは初日から完璧に**。競合(&Fanfunの公演情報内蔵)との差別化も、情報の網羅性ではなく通知・管理の信頼性で取る。

### F2. 公演・予定管理

- 公演(イベント)の登録: タイトル・推し・種別(ライブ/舞台/リリイベ/配信/その他)・会場・複数日程
- 月カレンダー/リスト表示。当落ステータスがカレンダー上で色分け(申込中=グレー、当選=推し色、落選=薄灰)
- 公演当日の準備通知(前日21:00: チケット/身分証リマインド)

### F3. 申込管理(当落ライフサイクル)= コア機能

**ステータス遷移**:

```
[申込予定] → [申込済(当落待ち)] → [当選] → [入金待ち] → [入金済] → [発券待ち] → [発券済] → 参戦
                    ↓                                ↓期限超過
                 [落選] → (復活当選待ち)          [期限切れ⚠]
```

- **申込登録フロー(3ステップ・30秒以内)**:
  1. 公演を選ぶ(または新規作成)
  2. 申込枠を選ぶ: プレイガイド/先行種別テンプレ(FC先行・プレ先行・ぴあ先行・ローチケ先行・イープラス先行・一般・リセール正規)
  3. 名義・希望公演(第1〜第N希望)・枚数・当落発表日・入金期限ルールを確認して保存
- **テンプレシステム**: 先行種別ごとに「発表日の初期値(申込締切+N日)」「入金期間(発表+N日)」を持つ。**すべてユーザー編集可能**(プレイガイドの仕様変更に強い設計)。テンプレ自体の追加・編集もできる
- **名義×公演のマトリクスビュー**: 行=名義、列=公演日で当落状況を一覧(重複当選・全滅リスクの見える化)
- 当落結果の記録はワンタップ(発表日通知から直接「当選/落選」を選べるアクション付き通知)
- 当選時: 座席情報・支払額を追記 → 入金期限通知が自動セット、収支(F4)に支出予定として自動反映
- 申込ページURLを保存でき、アプリ内ブラウザで開ける(v1.2でアフィリエイトリンク化)

### F4. 収支・予算

- 支出記録: 金額・カテゴリ(チケット/グッズ/遠征交通/宿泊/配信課金/その他)・推し・日付・メモ。当選チケットは自動で支出予定に計上
- 月次予算: 全体上限+推し別上限(任意)。80%到達・超過で通知
- 集計: 月別・推し別・カテゴリ別のグラフ。「推しに使った総額」の累計表示
- 推し貯金(シンプル版): 目標額と積立記録、進捗バー

### F5. 通知(ローカル通知)

| 通知 | タイミング | 備考 |
|---|---|---|
| 申込締切 | 締切前日20:00・当日12:00 | [申込予定]のまま放置されたもの |
| 当落発表 | 発表日時ちょうど | アクション付き(当選/落選をその場で記録) |
| 結果未記録フォロー | 発表日21:00 | ステータス未更新の場合のみ |
| **入金締切** | **前日20:00・当日9:00** | 最重要。当日通知は「今日中!」の強い文言 |
| 発券開始 | 開始日10:00 | コンビニ発券枠 |
| 一般発売 | 発売10分前 | 任意登録 |
| FC更新 | 7日前・前日 | F1から自動 |
| 公演前日 | 前日21:00 | チケット・身分証・持ち物 |

- すべて**ローカル通知**(スケジュール登録)。プッシュサーバー不要=運用費ゼロ
- アプリ起動時に全通知スケジュールを再計算(データ変更との整合)
- 通知ごとにオン/オフ・時刻カスタマイズ可(設定画面)

### F6. 当選シェアカード

- 当選記録時に「当選おめでとうカード」を自動生成: 推し色グラデ背景+公演名+日付+(任意で)名義数/申込数の戦績。デザインテンプレ無料2種/Pro8種+
- 画像保存・OSシェアシート経由でX/Instagramへ。**カード右下に小さくアプリ名ロゴ**(オーガニック獲得の導線)
- 個人情報(名義名・座席詳細)はデフォルトで非表示(晒し事故防止)

### F7. Pro課金(サブスクリプション)

| | 無料 | Pro |
|---|---|---|
| 推し登録 | 3人 | 無制限 |
| 名義 | 2つ | 無制限 |
| 同時進行の申込 | 15件 | 無制限 |
| シェアカードテンプレ | 2種 | 全種+推し色カスタム |
| データエクスポート(CSV) | ✕ | ◯ |
| 手動バックアップ/復元 | ◯ | ◯(将来: クラウド自動) |
| 価格 | — | **月400円 / 年3,800円**(7日間無料トライアル) |

- 実装はRevenueCat(月$2,500売上まで無料)。App Store IAP(Small Business Program 15%)
- **コア機能(当落管理・通知・収支)は無料で完結**させる。無料枠制限はライト層が困らず、ヘビー層(ペルソナ①)だけが上限に当たる設計
- MVPでは広告なし(レビュースコア優先)。広告導入はv1.1で数値を見て判断

### F8. 設定・その他

- テンプレ編集、通知設定、テーマ(ライト/ダーク)、データの手動バックアップ(ファイル書き出し)/復元
- 利用規約・プライバシーポリシー・特商法表記(ストア内課金のため App Store は Apple が販売主体)・お問い合わせ(Googleフォーム)

## 4. 非機能要件

- **ローカルファースト**: 全データは端末内SQLite。アカウント登録不要で即利用開始(オンボーディング離脱を最小化)。ネットワークなしで全機能動作
- **プライバシー**: 個人データを外部送信しない(分析イベントは匿名・最小限)。「推し活データは端末の外に出ません」を訴求点にする
- パフォーマンス: 起動2秒以内、申込登録30秒以内で完了する導線
- 信頼性: 通知は端末再起動後も維持(OSのスケジュール通知仕様に準拠)。データ破損対策として自動ローカルバックアップ(直近3世代)
- アクセシビリティ: Dynamic Type対応、推し色使用時のコントラスト自動補正
- 対応OS: iOS 16+(MVP)。Android(v1.1以降、Expoなので追加コスト小)

## 6. 画面構成

```
タブ構成:
├─ ホーム        … 直近の期限(入金>発表>締切の優先順)、今週の予定、今月の予算残
├─ 当落          … 申込一覧(ステータス別セグメント)/名義×公演マトリクス切替、[+]申込登録
├─ カレンダー    … 月表示+リスト、公演・発表日・締切がプロット
├─ 収支          … 今月のサマリー、支出一覧、グラフ、予算・貯金
└─ 推し          … 推し一覧(カード)、FC更新、設定への導線

モーダル/詳細:
・申込登録ウィザード(3ステップ) ・申込詳細(ステータス変更・当選記録)
・当選シェアカード生成 ・公演詳細 ・推し編集 ・Paywall ・設定/テンプレ編集
```

- オンボーディング(初回起動): ①推しを1人登録(名前+色だけ) → ②「直近で申し込む予定は?」で申込登録を体験 → ③通知許可リクエスト(価値を見せた後に求める)

## 7. 技術選定

| 領域 | 選定 | 理由 |
|---|---|---|
| フレームワーク | **Expo(React Native)+ TypeScript** | ローカル通知・IAP・ウィジェット(expo-widgets)が揃う。React系はバイブコーディングの学習データが最厚。iOS/Android両対応 |
| ルーティング | expo-router | ファイルベースで見通しが良い |
| DB | expo-sqlite + Drizzle ORM | ローカルファースト。型安全なスキーマ管理 |
| 状態管理 | Zustand(+ SQLiteクエリフック) | 小規模に十分 |
| 通知 | expo-notifications(ローカルスケジュール) | サーバー不要 |
| 課金 | RevenueCat(react-native-purchases) | サブスク実装の定番、$2,500/月まで無料 |
| シェアカード | react-native-view-shot + expo-sharing | View→画像化 |
| グラフ | victory-native または軽量自作 | 収支画面 |
| 分析 | 匿名・最小限(Aptabase等のプライバシー配慮型) | 「データは端末の外に出ない」訴求と両立する範囲 |
| ビルド/配布 | EAS Build + TestFlight | ソロ開発の標準 |

- サーバーサイド: **MVPでは持たない**(ランニングコスト0円)。将来のクラウド同期はSupabase想定
- 必要アカウント: Apple Developer Program(年99ドル/約15,800円)← 唯一の固定費

## 8. 法務・ストア対応

- 利用規約: 転売目的の利用禁止を明記、譲渡マッチング等の機能は提供しない旨のポリシー
- プライバシーポリシー: ローカル保存中心・分析は匿名最小限、広告SDKなし(MVP)
- 特商法表記: App Store経由の課金はAppleが販売主体。自社サイト販売を始める場合に整備
- チケット不正転売禁止法: 本アプリは個人の申込管理のみで抵触しないが、機能追加時のレッドライン(転売価格の記録・出品支援はしない)として文書化
- 商標: 正式名称決定時に商標検索(「当落ノート」他候補)。「ぴあ」等の商標をアプリ名・スクショで使わない

## 10. リスクと未決事項

- 支払い意欲リスク → コア無料+Pro上限設計で対応(§F7)。転換率が1%を切る場合はアフィリエイト・広告へ軸足
- Oshibanaの管理機能参入 → スピード優先。「期限とお金」の実データが貯まるロックインを先に作る
- **未決事項**: ①正式なアプリ名(商標チェック込みで3案出して選定) ②iOS先行で問題ないか(ターゲット層のiPhone比率は高い) ③Android版の時期
