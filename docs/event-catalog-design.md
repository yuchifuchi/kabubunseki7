## 1. ゴール(ユーザー発案の体験)

1. 推しを登録すると、その推しの**公演名(会場込み)が自動で候補表示**される
2. **公演日はプルダウン**から選ぶだけ(複数日程のツアーに対応)
3. 先行枠(FC先行/ぴあ等)を選ぶと、締切・当落発表日も自動プリフィル
4. 申込登録後、**実際の申込ページ(FC先行ならお知らせページ、ぴあならぴあの公演ページ)へワンタップで遷移**
5. これらのURL・公演情報は**日々自動更新**される

## 2. 調査で判明した前提(ここは動かせない)

### ❌ できないこと・やらないこと

- **プレイガイド(ぴあ/イープラス/ローチケ)本体のスクレイピングは不可能かつ危険**。実測でも3社ともデータセンター経由のアクセスをWAFで一律403遮断(robots.txtすら返さない)。BOT対策条項も強化されており、回避策(住宅IPプロキシ等)を使うのは規約違反+偽計業務妨害リスク(岡崎図書館事件の教訓)を自ら踏みにいく行為 → **絶対にやらない**
- **FCログイン内の情報は機械収集不可**(会員規約違反)。FC先行の申込期間はFC内のみ掲載のケースが多い → 別の方法で埋める(後述)
- 既存サービス調査でも「第三者サイトの無断クロールで成立しているサービスは存在しない」(イープラス/ぴあ公式アプリ=自社データ、LiveFans/イベンターノート=ユーザー投稿型)

### ✅ 使える正規ルート

| ソース | 内容 | コスト・条件 |
|---|---|---|
| **音楽ナタリーのアーティスト別RSS** | 公演発表・チケット情報のニュース。推し単位で購読可能 | 無料。最有力 |
| SPICE(イープラス系メディア)RSS | 同上 | 無料 |
| アーティスト公式サイトのNEWS/LIVEページ | 一次ソース。オフィシャル先行・プレイガイド先行は公開される | robots.txt遵守・1日1〜2回・低負荷で差分監視 |
| schema.org/Event(JSON-LD) | 一部の公演ページ・会場サイトに埋め込みあり | HTMLパースより壊れにくい |
| X API(公式アカウント監視) | 2026年から従量課金化。数十アカウントなら月数ドル | Phase 2。画像のみの告知はLLMビジョンで読む |
| ぴあWEB API / ARTIST COMMONS | 4大プレイガイドの公演・発売データを正規取得できる唯一の経路 | 法人・有料/会員制。**サービスが育ったら交渉**(ベンチャー実証に柔軟対応と明記あり) |

### 法的ガードレール(要件に明記する)

- 公演名・日付・会場・発売日は「事実」であり著作権の対象外 → **事実の構造化データのみ扱う**。告知文の転載・画像の複製はしない。必ず出典リンクを付ける
- 特定サイトのデータベースの「デッドコピー」は不法行為リスク → 複数ソースからの事実抽出に限定
- アクセスは robots.txt 遵守・低頻度・逐次・User-Agent明示(連絡先入り)

## 3. 提案アーキテクチャ: 3層ハイブリッド収集

```
[Tier A] 自動収集(安全ルートのみ)                     [毎日1〜2回]
  ナタリーRSS / SPICE RSS / 公式サイトNEWS / JSON-LD
      → LLMで構造化抽出(公演名・会場・日程・先行種別・期間・URL)
      → 検品キュー(管理画面で人間が承認 ※初期は必須)
      → 公演カタログDBへ公開(出典リンク・確度ラベル付き)

[Tier B] 公式X監視(Phase 2)                          [準リアルタイム]
  推し対象アカウントの新着ポスト → LLM解析(画像OCR込み) → 検品キューへ

[Tier C] ユーザー補完(FC先行の穴を埋める)
  ① 貼り付け解析: FCのお知らせ文/申込確認メールを貼り付け → 端末上でLLM解析 → 申込情報を自動入力
  ② UGC共有(Phase 3): 「同じ推しのユーザーが登録した先行情報」を通報・検品付きで共有
```

- アプリ側: 推し登録時にカタログのアーティストと紐付け(いなければ従来どおり手入力+「この推しの情報も集めて」リクエストボタン)
- 申込フォーム: カタログから公演選択 → 日程プルダウン → 先行枠選択で締切・発表日プリフィル → 保存後「申込ページを開く」(ぴあ・ローチケはアフィリエイトリンク化)
- **重要な設計原則: カタログはあくまで「提案」で、全フィールドは手修正可能**。カタログが無い推しでも今のM1の手入力フローがそのまま使える(機能が縮退しても壊れない)

## 4. 正確性の担保(このアプリの生命線)

締切情報を間違えると「入金忘れを防ぐアプリ」への信頼が死ぬ。よって:

1. 初期は**全件人間検品**(管理画面で承認するまで公開されない)。1日5〜10分想定
2. 表示には必ず**出典リンク+「申込前に公式でご確認ください」**を添える
3. 確度ラベル: 「公式確認済み」(検品済み)/「自動抽出」(Phase 2で検品省略する場合)
4. ユーザーの「情報が違う」報告ボタン → 即時修正フロー
5. リンク切れはユーザー報告ベース(プレイガイドへの機械的な死活チェックは不可のため)

## 5. インフラ案(2択)

| | 案X: GitHub Actions + 静的JSON配信 | 案Y: Supabase |
|---|---|---|
| 構成 | 日次のActions cronでパイプライン実行 → 抽出結果をPRとして作成 → **PRレビュー=検品** → マージでCDN(Cloudflare Pages)に公開 | DB+管理画面+API。検品は管理画面 |
| コスト | ほぼ0円(LLM API 数百円/月のみ) | 無料枠〜$25/月 |
| 検品体験 | GitHubのPR画面(差分が見やすい) | 専用管理画面(作る手間あり) |
| UGC拡張 | 不向き(Phase 3でSupabase移行) | そのまま拡張可能 |
| 推奨 | **Phase 1はこちら**(運用がGitで監査可能、サーバー管理ゼロ) | Phase 3(UGC)で移行 |

- LLMコスト試算: 対象100アーティスト×RSS/NEWS差分 → 1日数十〜百記事の抽出で**月数百円〜千円程度**(安価モデル+キャッシュ)
- アプリはカタログJSONを起動時にフェッチ(アーティスト単位・CDNキャッシュ)。オフライン時は最終取得分を使用 → ローカルファースト原則は維持

## 6. カバレッジ戦略(要決定)

- 全ジャンル網羅は運用が破綻する。**初期はパイロットとして1〜2ジャンル×50〜100組**に絞り、精度と運用負荷を確認してから拡大
- 推し登録時にカタログ外の推しは「リクエスト」を送れる → リクエスト数順にソース(公式サイトURL・ナタリーID・Xアカウント)を登録して対象拡大
- ジャンル特性: 男性アイドル/K-POPはFC先行中心(Tier Cの比重大)、声優・アニソン/バンドは公開先行が多くTier Aで拾いやすい

## 7. 実装フェーズへの影響(マイルストーン再編成案)

| 旧 | 新(提案) | 内容 |
|---|---|---|
| M2: 収支・カレンダー | **M2: カタログ体験(アプリ側)** | 公演サジェスト・日程プルダウン・申込URL遷移・貼り付け解析。カタログは最初**手動整備の20〜50組**で開始(パイプラインを待たずに体験を完成させる) |
| M3: シェアカード・課金 | **M3: 収集パイプライン** | RSS+公式サイト+LLM抽出+PR検品フローの自動化。対象を50→100組へ |
| — | M4: 収支・カレンダー | 旧M2 |
| — | M5: シェアカード・Pro課金・公開準備 | 旧M3 |

- 理由: カタログはこのアプリ最大の差別化(競合の当落管理アプリはすべて手入力)なので前倒し。ただし**パイプラインより先にアプリ側の体験を手動データで完成**させ、「データを日々作る価値があるか」を早く確認する

## 8. リスクと未決事項

| リスク | 対応 |
|---|---|
| 誤情報で入金忘れ等の実害 | §4の検品+出典+免責。利用規約に「情報は参考であり正確性を保証しない」を明記 |
| 運用が続かない(毎日の検品) | 検品5〜10分/日に収まる規模から開始。負荷が超えたら対象を減らす勇気 |
| ソース側の仕様変更 | RSS優先(構造安定)。抽出失敗はアラートのみで落とさない |
| カバレッジへの不満 | 「リクエスト」機能で期待値をコントロール。カタログ外でも手入力で全機能使える |

### 壁打ちしたい論点(未決)

1. **初期ジャンル**: どこから始めるか(候補: ①男性アイドル/K-POP=市場最大だがFC中心、②声優・アニソン=公開先行多く自動化しやすい、③バンド/フェス=ナタリー網羅度高い)。あなた自身が詳しいジャンルから始めるのが検品品質の面でも有利

4. **マイルストーン再編**: §7の順序(カタログを収支より前倒し)で良いか
5. **X API課金**: Phase 2で月数ドル〜の従量課金を許容するか
